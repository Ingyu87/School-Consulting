# 고일초 컨설팅 문서 생성 웹앱 디자인 시스템

## 1. 디자인 방향

기존 `DESIGN-tesla.md`의 테슬라식 풀스크린 쇼룸 스타일은 이번 목적과 맞지 않는다. 새 웹앱은 `심층면담지.pdf`와 `운영계획서.pdf`의 행정 문서 스타일을 따르는 업무형 작성 도구여야 한다.

디자인 목표는 다음과 같다.

- 학교 담당자에게 신뢰감 있는 공식 문서 분위기 제공
- 입력 중인 내용이 최종 문서의 어느 위치로 들어가는지 직관적으로 이해 가능
- 진단 결과 시각화는 깔끔하되 과장 없이 문서형으로 표현
- PDF 양식의 남색 헤더, 얇은 표, 회색 구분 셀, 넓은 여백을 웹 UI에 반영
- 슬라이드 감성보다 `작성 중인 공식 서류` 느낌을 우선

## 2. 시각 기준 출처

### 2.1 운영계획서.pdf

주요 특징:
- 흰 배경
- 중앙 정렬 제목
- 얇은 검정 테두리 박스
- 남색 섹션 제목
- 표 헤더는 진한 남색
- 표 본문은 흰색/연한 회청색 교차 배경
- 하단에 기관 로고 푸터
- 문서 본문은 명조 계열에 가까운 차분한 글꼴

### 2.2 심층면담지.pdf

주요 특징:
- 상단 우측 프로젝트 문구
- 얇은 파란 라인
- 제목과 본문 구획이 명확함
- 체크박스 형식의 선택지
- 입력칸 중심의 큰 표
- 문서 페이지 번호와 푸터

## 6. 컴포넌트

### 6.1 DocumentPage

문서 한 페이지처럼 보이는 컨테이너.

스타일:
- 흰 배경
- 24~32px 내부 여백
- 얇은 회색 테두리
- 페이지 상단에 제목
- 필요 시 하단 푸터

### 6.2 SectionHeader

PDF의 `Ⅰ.`, `Ⅱ.`, `Ⅲ.` 제목 스타일을 따른다.

```text
Ⅰ. 우리학교 디지털 기반 교육 현황 알아보기
```

스타일:
- 남색 텍스트
- 아래 얇은 남색 라인
- 상하 여백 넉넉하게

### 6.3 FormTable

심층면담지/운영계획서의 핵심 입력 컴포넌트.

스타일:
- `border-collapse: collapse`
- 헤더 배경 Deep Navy
- 헤더 글자 흰색
- 구분 셀 배경 Table Gray
- 본문 셀 흰색
- 테두리 Border Gray

### 6.4 CheckListRow

PDF 체크박스 선택지를 웹에서 구현.

상태:
- 미선택
- 선택
- 확인 필요

### 6.5 ModuleCard

모듈 선택용 카드지만 카드형 장식보다 표 느낌을 유지한다.

내용:
- 모듈 번호
- 필수/선택
- 대상
- 차시
- 방향성
- 내용 예시
- 선택 여부

스타일:
- 선택 시 남색 좌측 바
- 필수 모듈은 잠금 표시
- 오류가 있으면 상단 작은 빨간 배지

### 6.6 ValidationMessage

PDF 기준 위반/주의를 표시한다.

예:
- `오류: 전체 연수는 총 12차시 이상 구성해야 합니다.`
- `오류: 모듈1/2/3 합계는 최대 5차시까지 가능합니다.`
- `주의: 학부모 과정은 식사·다과 제공 대상이 아닙니다.`

## 7. 진단 시각화 스타일

### 7.1 과정별 점수 그래프

- 가로 막대그래프
- 5점 만점 기준
- 단계 색상은 과장하지 않고 남색 계열 농도 차이로 구분

단계:
- 도약: 연한 남색 + 주의 배지
- 만족: 기본 남색
- 추월: 진한 남색

### 7.2 응답 분포표

운영계획서 Ⅰ장 표와 유사하게 표시한다.

열:
- 문항
- 응답
- 결과
- 비율

### 7.3 분석 요약

문장형 카드가 아니라 운영계획서에 들어갈 문단형 편집 박스로 제공한다.

예:

```text
고일초는 전반적으로 AI·디지털 교육 전환에 대한 이해도는 양호하나,
학생 기초 소양과 학부모 이해 영역에서 보완 요구가 확인된다...
```

## 8. 문서 다운로드 화면

### 8.1 출력 전 검토

다운로드 전 다음 상태를 보여준다.

- 심층면담지 필수 입력 완료 여부
- 운영계획서 필수 입력 완료 여부
- 연수 구성 PDF 기준 충족 여부
- 미확정 일정 목록
- 생성 문서 파일명

### 8.2 다운로드 버튼

버튼명:
- `심층면담지 다운로드`
- `운영계획서 다운로드`
- `전체 백업 저장`

버튼은 남색 계열로 통일한다. `docx`, `JSON` 같은 개발자식 라벨은 쓰지 않는다.

## 9. 문체 규칙

### 9.1 안내 문구

안내 문구는 PDF 공식 안내자료의 표현만 사용한다.

금지:
- 일정표 메모를 공식 기준처럼 표현
- 운영기관 확인 전 가능/불가를 단정
- 시간대만 보고 도시락 제공 여부를 판단

### 9.2 분석 문구

분석 문구는 `제안` 또는 `초안`임을 명확히 한다.

예:
- `분석 초안`
- `면담에서 확인 필요`
- `운영계획서 반영 전 검토`

### 9.3 일정 문구

CSV 일정은 항상 `가안` 또는 `초기값`으로 표시한다.

예:
- `고일초 일정 가안`
- `수정 가능`
- `확정 전 학교 확인 필요`

## 10. 반응형

데스크톱 우선 설계.

- 1280px 이상: 좌측 네비 + 문서 캔버스
- 768~1279px: 좌측 네비 축소
- 767px 이하: 상단 단계 탭 + 단일 컬럼

현장 노트북 사용을 우선하므로 1366x768에서도 주요 입력이 가능해야 한다.

## 11. 접근성

- 모든 입력에는 label 제공
- 오류 메시지는 색상뿐 아니라 텍스트로 표시
- 키보드 이동 가능
- 다운로드 버튼은 명확한 파일 목적을 이름에 포함

## 12. 진단 분석 대시보드 스타일

문서 작성 화면은 PDF 양식 스타일을 따르지만, 진단 분석 화면은 학교 담당자가 즉시 이해할 수 있도록 더 컬러풀한 데이터 대시보드 스타일을 적용한다.

이 영역의 목표는 다음과 같다.

- 학교의 현재 상태를 30초 안에 설명 가능
- 낮은 영역과 높은 영역이 즉시 구분됨
- 어떤 진단 결과가 어떤 연수 모듈로 연결되는지 시각적으로 이해됨
- 운영계획서에 넣을 분석 문단을 자연스럽게 도출

## 16. 애니메이션

### 16.1 원칙

- 분석 화면은 생동감 있게, 문서 입력 화면은 안정적으로
- 애니메이션은 정보 이해를 돕는 목적에만 사용
- 반복적으로 흔들리거나 튀는 효과 금지

### 16.2 적용 위치

- CSV 업로드 완료 후 카드 순차 등장
- 점수 숫자 count-up
- 막대그래프 width fill
- 도넛 차트 stroke animation
- 추천 모듈 연결선 fade-in
- 분석 초안 생성 시 highlight flash 1회

### 16.3 Motion Tokens

```css
:root {
  --motion-fast: 160ms;
  --motion-normal: 280ms;
  --motion-slow: 420ms;
  --motion-ease: cubic-bezier(0.2, 0, 0, 1);
}
```

### 16.4 접근성

`prefers-reduced-motion: reduce` 사용자는 모든 애니메이션을 즉시 완료 상태로 표시한다.

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.001ms !important;
    transition-duration: 0.001ms !important;
  }
}
```

## 17. CSV 업로드 UI

### 17.1 Empty State

처음 앱을 열면 다음 중 하나를 선택한다.

- `고일초 샘플 열기`
- `진단 CSV 업로드`
- `백업 파일 불러오기`

### 17.2 Upload State

업로드 중:
- 파일명 표시
- 파싱 단계 표시
- 분석 중 progress indicator

업로드 완료:
- 추출된 학교명 표시
- 추출된 문항/모듈 수 표시
- 파싱 경고가 있으면 노란 박스로 표시

### 17.3 Error State

CSV 구조를 찾지 못하면 다음을 제공한다.

## 18. 문서형 화면과 대시보드 화면의 연결

진단 대시보드에서 생성된 분석 초안은 운영계획서 화면으로 이어진다.

연결 방식:
- `운영계획서에 반영` 버튼
- 낮은 영역별 `면담 질문으로 추가` 버튼
- 추천 모듈 `연수 구성에 추가` 버튼

버튼명은 행동 중심으로 쓴다.

- `분석 초안 수정`
- `면담 질문에 추가`
- `추천 모듈 반영`
- `운영계획서로 보내기`
